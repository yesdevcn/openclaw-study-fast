# 第 8 章：常见踩坑与快速排查清单

再好的工具，只要涉及「安装 + 配置 + 网络 + 各种第三方服务」，就难免会踩坑。  
这一章我们不追求全面覆盖所有错误，只想帮你做到两点：

- 碰到问题不慌，大致知道属于哪一类；
- 有一套「通用排查步骤」，能自己先解决 70% 的小问题。

> 提示：具体版本变动、详细错误对照，请以官方文档和社区教程为准，本章更偏「思路与检查清单」。

---

## 8.1 大部分问题可以归为四类

- **安装与环境问题**
  - 安装脚本失败；
  - 某些命令找不到；
  - 依赖缺失、权限不足等。

- **网络与 API 问题**
  - 连不上模型服务；
  - 返回 401/403/429/5xx 等错误；
  - 墙与代理相关问题。

- **配置与权限问题**
  - 配置文件写错、漏写；
  - Gateway 启动失败；
  - Skills 没有足够的系统权限（读写文件、日历等）。

- **使用与预期不一致**
  - AI 行为「不听话」「乱用工具」；
  - 自动化任务不按预期触发或执行不全。

先判断自己大致落在哪一类，再针对性处理。

---

## 8.2 安装与环境问题：基本检查清单

常见症状：

- `openclaw` 命令不存在；
- 安装脚本中断、报权限错误；
- 某些系统依赖缺失（如 curl/git 等）。

排查步骤：

1. **确认系统版本与架构**
   - macOS / Linux / Windows 是否满足官方最低要求；
   - 对于 Windows，尽量在 WSL2 中安装。

2. **重新运行官方安装脚本**
   - 观察输出中的错误提示（不要只看最后一行）。

3. **检查 PATH 与权限**
   - 安装目录是否在系统 PATH 中；
   - 是否使用了受限用户、权限不够。

4. **参考官方/社区 FAQ**
   - 很多常见环境问题，官方文档和社区教程中都有专门章节节；
   - 可以搜索错误关键字 +「OpenClaw」来定位类似经验。

---

## 8.3 网络与 API 问题：看错误码就能猜一半

常见错误码含义（简化版）：

- **401 未授权**
  - API Key 错误、过期或没填；
  - Key 不匹配当前使用的模型/服务。

- **403 禁止访问**
  - IP 不在白名单；
  - 账号或项目没有权限调用该接口。

- **429 频率限制**
  - 请求速度过快、超出配额；
  - 某些中转/服务在高峰期会触发限流。

- **5xx 服务器内部错误**
  - 对方服务异常；
  - 或请求参数导致服务端处理异常。

排查建议：

1. 确认在本机/服务器上能直接访问模型服务的 API 地址；
2. 再三核对 API Key 是否填写正确、有无多余空格等；
3. 如使用代理/中转，确认配置文件中的 `base_url` 等字段无误；
4. 在错误日志中找到完整的请求/响应片段，交给 OpenClaw 帮你翻译解读。

---

## 8.4 配置与权限问题：Gateway 启不起来怎么办？

常见症状：

- 执行 `openclaw gateway start` 或 `restart` 后，立刻退出；
- 日志/终端内看到明显的 JSON 解析错误、字段缺失等；
- 版本升级后，旧配置不再兼容（例如必须显式设置认证方式）。

排查建议：

1. **检查配置文件格式**
   - 确保 JSON/YAML 无语法错误（可以用在线/本地校验工具）；
   - 对照官方示例，检查是否少写了必需字段。

2. **注意版本变更说明**
   - 某些版本会有 Breaking Change（如必须设置认证模式等）；
   - 遇到「升级后突然启动不了」，优先去看版本更新说明。

3. **逐步简化配置**
   - 先退回到最小可用配置（只保留一个模型提供商、一个 Agent）；
   - 确认能启动后，再逐段加回原有配置。

4. **系统权限相关**
   - 在 macOS 上，确保给了必要的「完全磁盘访问」「日历访问」权限等；
   - 在云端，确保配置文件目录对运行用户可读写。

## 8.5 Skills 与工具调用问题：为什么它「动不了手」？

常见情况：

- 明明装了某个 Skill，但 OpenClaw 从不调用；
- 调用后报「权限不足」「路径不存在」「服务连接失败」等。

排查思路：

1. **确认 Skill 已启用**
   - 在 Control UI / 配置文件中检查该 Skill 是否处于启用状态；
   - 有些 Skill 可能默认关闭，需要手动勾选或配置。

2. **检查 Skill 配置**
   - 必需参数是否都填了（比如 API Key、根目录路径等）；
   - 对路径特别敏感的 Skill，注意绝对/相对路径差异。

3. **看错误信息**
   - 很多 Skill 会在日志中给出比较直白的错误原因；
   - 可以复制错误文本交给 OpenClaw 解释，并请它给出修复建议。

4. **权限与安全限制**
   - 某些高危操作（如执行命令）可能需要特别开启；
   - 在云端环境中，还要考虑容器/系统层面的限制。

---

## 8.6 使用效果问题：AI「不听话」怎么办？

有时问题不在系统，而在「沟通」上：

- 回答风格不符合预期；
- 总是给出笼统一堆废话；
- 忽略你强调的约束条件。

改进建议：

1. **补充背景与约束**
   - 多说明「你是谁」「你要干什么」「有哪些前提」；
   - 给出更清晰的输出格式要求。

2. **反复迭代而不是重来**
   - 在同一会话里直接说：「刚才第 3 点不符合我的实际情况，请改成……」；
   - 让它在已有内容上修正比从头来更高效。

3. **固定常用模板**
   - 对于经常用到的任务型指令，整理成固定模板；
   - 每次只修改变量部分，稳定性会高很多。

---

## 8.7 求助他人时，怎样描述问题更有效？

无论是向社区还是同事求助，尽量做到：

- 说清楚你在 **做什么**：
  - 环境（本地/云端，系统版本）；
  - 使用的 OpenClaw 版本；
  - 想实现的目标。

- 附上关键证据：
  - 执行的命令；
  - 完整或裁剪后的错误输出；
  - 相关配置（去除敏感信息）。

- 说明你已经尝试过什么：
  - 看过哪些文档/章节节；
  - 做过哪些排查（例如重启、简化配置）。

这样别人更容易在短时间内给出有价值的建议，也能帮助后来的用户少踩同样的坑。

本章的目的，不是让你记住所有错误细节，而是帮你建立一个心态：

> **出问题很正常，有一套方法去拆分和排查就行。**

真正遇到具体错误时，你可以：

- 先用这一章的分类和清单做「自救」；
- 再结合官方文档和社区教程查细节；
- 实在解决不了，再带着充分信息向他人求助。
