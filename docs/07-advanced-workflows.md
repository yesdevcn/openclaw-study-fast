# 第 7 章：进阶玩法——多 Agent 与自动化工作流

前面几章，我们更多是在「你开口、它执行」的一问一答模式下使用 OpenClaw。  
这一章，我们把视角抬高一点，看看它如何：

- 通过 **多 Agent** 分工合作；
- 通过 **自动化工作流** 在你不在线时持续工作。

不追求一次看懂全部，只要建立基本认知即可。

---

## 7.1 什么是多 Agent？

简单理解：

- 一个 Agent = 一组「人设 + 能力 + 配置」的 AI 助理；
- 多 Agent = 你同时拥有多位性格与专长不同的助理。

典型例子：

- 一个偏「总助理」：帮你统筹任务、做时间管理；
- 一个偏「内内容创作」：帮你写文章节、做排版、想标题；
- 一个偏「技术」：帮你看代码、查 Bug、设计架构；
- 一个偏「资讯/研究」：帮你盯行业新闻、汇总情报。

在 OpenClaw 里，多 Agent 通常意味着：

- 在配置中为不同 Agent 定义不同的：
  - 默认模型；
  - 可用 Skills 集合；
  - 工作空间（访问哪些文件、知识库等）；
- 在不同入口/机器人上绑定不同 Agent，或在同一入口内部切换 Agent。

---

## 7.2 为什么要用多 Agent？

从实践角度看，多 Agent 主要带来三点好处：

- **分工更清晰**
  - 每个 Agent 专注做自己擅长的事情；
  - 提示词、人设、默认工具都围绕一个主题优化。

- **风险与权限隔离**
  - 技术向 Agent 可能需要更高的系统权限（读写代码库、执行命令等）；
  - 日常聊天 Agent 则可以更「安全保守」；
  - 避免一个 Agent「什么都能做」，增加误操作风险。

- **体验更稳定**
  - 每个 Agent 的提示词不会互相干扰；
  - 对你来说，也更容易建立使用习惯：「写代码找谁」「写文案找谁」。

---

## 7.3 一个简单的多 Agent 配置思路（概念级）

不写具体 JSON，只讲思路，方便你之后对应到官方配置文档：

- 定义 `assistant-main`
  - 角色：总助理 / 生活工作综合助手；
  - 模型：适合日常对话的通用模型；
  - Skills：文件（只读为主）、日历、待办、网页抓取、简单通知。

- 定义 `assistant-coding`
  - 角色：编程助手 / 代码审查；
  - 模型：偏代码场景的模型；
  - Skills：代码库访问、Issue/PR 管理相关、命令执行（如果你接受风险的话）。

- 定义 `assistant-content`
  - 角色：内内容创作者；
  - 模型：擅长长文案、创意写作的模型；
  - Skills：网页抓取、知识库访问、排版与导出相关工具。

然后：

- 在不同 IM 机器人或不同入口上分别绑定对应 Agent；
- 或者在 Control UI 中给每个 Agent 单独开一个「空间」。

具体配置语法请以官方文档和你参考的进阶教程（如 `awesome-openclaw-tutorial` 中的多 Agent 章节节）为准。

## 7.4 什么是自动化工作流？

前面的例子多是「你说一句，它干一次」。  
自动化工作流则更多强调：

- **触发条件**：什么时候开始？
  - 定时（每天 9:00）；
  - 事件（某个网站更新、某个文件变化、某个 API 返回特定值）；

- **一串具体步骤**：每一步做什么？
  - 抓数据 → 分析 → 生成报告 → 发通知；

- **最终输出**：结果去哪儿？
  - 你常用的 IM 群组；
  - 邮件；
  - 存入某个文件/知识库。

OpenClaw + Skills 的组合，非常适合作为这类工作流的「大脑」和「调度中心」。

---

## 7.5 一个典型案例：每日 AI 行业简报

需求：  
你希望每天早上 9:00 收到一份「简洁的 AI 行业简报」，发到某个飞书/企微群里。

思路：

1. **触发方式**
   - 使用定时任务相关的能力（可能由某个自动化 Skill 或外部定时系统触发）；

2. **数据来源**
   - 访问若干新闻网站、RSS、API；
   - 或读取你自己维护的一份「关注源列表」文件。

3. **处理步骤**
   - 把过去 24 小时的相关新闻抓取回来；
   - 过滤掉重复与低质量内容；
   - 用模型生成简要摘要 + 影响点评；
   - 按重要程度排序。

4. **输出**
   - 生成一条结构化消息（标题 + 3–5 条要点 + 链接）；
   - 通过 IM 集成类 Skill 发到指定群组。

在配置好之后，你每天可以什么都不用做，照样在上班前 5 分钟拿到一份「订制版简报」。

## 7.6 一个典型案例：代码库「守夜人」

对于开发者团队，可以让某个技术向 Agent 当「守夜人」：

需求：  
当仓库里有新的 PR / Issue / 重要报错时，希望有一个 Agent 自动：

- 初步查看内容；
- 给出简单分析和建议；
- 把关键信息推送到团队沟通工具。

思路：

1. 通过 GitHub 等平台的 Webhook / API 监听变更；
2. 将事件内容交给 `assistant-coding` 这类技术 Agent；
3. 让它：
   - 阅读 PR/Issue 内容；
   - 标记潜在风险点；
   - 给出一段「可供人类快速决策」的摘要；
4. 使用 IM/邮件 Skills 把结果发到指定频道。

在这样的自动化下，团队不用时时盯仓库，也能尽早发现问题和机会。

## 7.7 如何循序渐进地玩转这些进阶能力？

建议不要一口气上「多 Agent + 复杂自动化」。可以按以下节奏：

1. **先跑通 1 个 Agent + 1 个简单定时任务**
   - 例如：每天整理一份「个人待办 + 重要邮件摘要」；

2. **再拆分出 2–3 个职责不同的 Agent**
   - 把目前散落的任务，归类到对应 Agent 名下；

3. **最后再考虑跨 Agent 的协同与更复杂的工作流**
   - 例如：`资讯 Agent` 收集信息 → `总助理 Agent` 帮你做决策建议。

过程中，始终记住一个原则：

> **只要现有流程已经在帮你省时间，就先稳定用一段时间，再考虑下一层抽象。**

本章你需要带走的是这样一幅图景：

- OpenClaw 不只是一个助手，而是可以容纳多种角色的「AI 团队」；

- 等你准备好时，可以一点点把自己的工作方式迁移、封装进这套系统里。
